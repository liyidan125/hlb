FE Credit 数据平台建设现状调研提纲
FE Credit Data Platform Status — Due-Diligence Questionnaire

调研目的 / Objective

以微众银行（WeBank）零售信贷数据体系作为参考架构（benchmark），系统摸清 FE Credit 当前数据平台的现状、能力成熟度与业务支撑情况，进而诊断其核心痛点与短板，为后续平台建设/优化方案提供输入。

调研方法 / Methodology

差距分析（Gap Analysis）—— 围绕"数据战略—架构—资产—开发—分析—质量—AI—安全—业务赋能"九大维度，每个维度先摸现状，再挖痛点，并与微众对标基线对照定位成熟度。

FEC 已知背景 / Known FEC Context

FEC 数字客户平台已部署于 AWS 公有云 (包含 Data Lake 等)。核心银行系统(T24等)位于 VPB 本地数据中心。即 FEC 已是 "AWS 公有云 Data Lake + 本地数据中心核心系统"的混合架构。

维度一：数据战略、组织与规划 / Dim 1: Data Strategy & Org
诊断意图 / Diagnostic intent：判断数据工作是否"有顶层设计、专职团队、持续投入"，还是处于"需求驱动、零散建设、无人统筹"的原始状态。
1. FE Credit 是否有明确的数据战略/数据中台规划？由哪个部门（CDO/IT/业务）牵头？数据团队当前人员规模（开发、数据工程、分析师、算法）？
EN: Does FE Credit have an explicit data strategy / data-middleware roadmap? Which function leads it? What is the current headcount of the data team?
2. 数据需求目前是如何产生与排期的——业务提需求、IT 接单的"项目制"，还是有统一的数据产品规划？
EN: How are data demands generated and prioritized today — a "project-based" model, or a unified data-product roadmap?
3. 过去 2–3 年在数据平台上的主要投入与建成了哪些系统？最大的几笔投入回报如何（业务方是否真正用起来）？
EN: What were the major data-platform investments and systems built in the past 2–3 years? What was the ROI of the largest ones?
4. 当前数据工作最大的阻力和矛盾是什么——预算、人才、跨部门协同，还是业务方的信任与使用意愿？
EN: What are the biggest frictions/contradictions in data work today?
5. 监管（SBV 越南国家银行、个人信息保护）对数据治理/数据出境/本地化有哪些硬性要求？是否已成为建设的强制驱动力？
EN: What hard regulatory requirements exist around data governance / cross-border transfer / localization?
维度二：数据架构、数仓分层与 Data Lake 现状 / Dim 2: Data Architecture & Lake
诊断意图 / Diagnostic intent：判断 Data Lake 之上是否建立了规范的数仓分层与主题域，还是退化为"数据沼泽（data swamp）"。
6. AWS Data Lake 当前存储了哪些数据？数据格式与分区/分层策略如何？数据规模与增长？是否已出现"数据沼泽"？
EN: What data currently lives in the AWS Data Lake? Data formats and partitioning/zone strategy? Scale and growth? Has it become a "data swamp"?
7. Data Lake 之上是否已建数仓分层（ODS → DWD → DWS → ADS）？还是尚无分层加工与规范化建模？分层是否真正执行？
EN: Is a warehouse layering built on top of the Data Lake? Or is it still only a raw Data Lake? Is the layering actually enforced?
8. 数据来源与入湖：VPB 本地数据中心的核心数据（T24 Core 等）是否同步入湖？跨数据中心→云的集成链路如何？
EN: Sources & ingestion: Is the on-prem VPB core data replicated into the lake? What is the on-prem→cloud integration link?
9. 主题域如何划分？是否覆盖信贷全流程？跨产品口径是否统一？
EN: How are subject areas partitioned? Do they cover the full credit lifecycle? Are cross-product definitions unified?
10. 同一指标在云上 Data Lake、本地系统、各报表间口径是否一致？以哪一侧为准（source of truth）？分歧如何仲裁？
EN: Is the same metric consistent across the cloud Data Lake, on-prem systems and various reports? Which side is the source of truth?
维度三：AWS 云上计算/存储与集成 / Dim 3: AWS Compute & Integration
诊断意图 / Diagnostic intent：判断 Data Lake 的存储与计算选型是否合理，以及 AWS 云 ↔ VPB 本地之间的集成是否顺畅。
11. Data Lake 的存储与计算分别用了哪些 AWS 服务（S3, EMR, Athena, Redshift 等）？是否有明确选型理由与架构治理？
EN: Which AWS services are used for Data Lake storage & compute? Is there a clear rationale and architecture governance?
12. 是否在 S3 之上建了云数仓（Redshift）或湖仓一体？Redshift 与 S3 是分离存储还是联邦查询？是否多套引擎并行导致运维复杂？
EN: Is a cloud warehouse (Redshift) or lakehouse built on top of S3? Are they separately stored or federated? Are redundant engines causing complexity?
13. 实时计算能力如何——是否具备毫秒/秒级流处理？风控实时决策等是否已上线？实时与离线口径如何保持一致？
EN: Real-time compute capability — is there ms/s-level stream processing? Are real-time risk decisions live? How is consistency maintained?
14. 混合云数据集成：AWS 与 VPB 之间数据如何同步（专线 / VPN）？核心系统 CDC 是否准实时入湖？网络带宽与稳定性如何保障？
EN: Hybrid-cloud integration: how is data synced between AWS and the on-prem VPB DC? Are core-system CDC ingested in near-real time?
15. 调度与可观测性——跑批用什么调度？数据链路监控、产出 SLA、失败告警与重试机制？近半年是否有数据事故？
EN: Orchestration & observability — what schedules the batches? Pipeline monitoring, SLA, alerting? Any recent data incidents?
维度四：数据资产管理体系 / Dim 4: Data Asset Management
诊断意图 / Diagnostic intent：判断数据是否"可见、可找、可用、可管"，还是"重复建设、找数难、无台账"。
16. FE Credit 是否有数据资产目录/元数据管理平台？全行数据资产是否已盘点？
EN: Does FE Credit have a data-asset catalog / metadata-management platform? Has a bank-wide inventory been done?
17. 数据血缘（从源头到报表的字段级血缘）是否已建立？影响分析能否做到？
EN: Is field-level data lineage established? Can impact analysis be performed?
18. 指标/标签是否有统一的命名规范、业务定义、Owner 与审批流程？是否存在大量口径冲突？
EN: Do metrics/tags have unified naming conventions, definitions, owners and workflows? Are there definition conflicts?
19. 数据资产的使用情况是否可度量？是否有资产下发的服务化机制？
EN: Is asset usage measurable? Is there a service-based asset-distribution mechanism?
20. 当前"找数、懂数、用数"的体验如何——平均要多久？主要卡点在哪？
EN: How is the "find/understand/use data" experience today? Where is the main bottleneck?
维度五：标签与风控变量加工平台 / Dim 5: Tag & Risk-Variable Platform
诊断意图 / Diagnostic intent：判断风控变量/标签是"集中化、配置化、可复用"，还是"硬编码、重复造轮子、上线慢"。
21. 风控变量（features）/标签目前是如何开发的——纯代码 IT 开发，还是业务可自助配置？是否存在低代码能力？
EN: How are risk variables developed today — pure code by IT, or self-service config? Is there low-code capability?
22. 从一个新变量"提出需求→开发→验证→上线"平均要多久？瓶颈在哪？
EN: Average time from a new variable "request → develop → validate → go-live"? Where is the bottleneck?
23. 变量加工是否有可复用的算子库？是否有图谱变量加工能力与关系网络数据建设？
EN: Is there a reusable operator library? Is there graph-variable engineering and relationship-network data?
24. 变量的"分析态"与"生产态"是否分离？上线前是否有样本比对验证机制？
EN: Are the "analysis mode" and "production mode" of variables separated? Is there sample-comparison validation?
25. 变量/标签是否做到"一处定义、多处使用"？还是口径漂移？支持不同时效灵活联结吗？
EN: Do variables achieve "define once, use everywhere"? Timeliness management supported?
维度六：外部数据源管理 / Dim 6: External Data Source Management
诊断意图 / Diagnostic intent：判断外部数据是"统一接入、标准化、可计费、稳定可控"，还是"字段混乱、成本失控"。
26. 目前接入了多少家外部数据源？接入新数据源平均要多久？流程是否配置化？
EN: How many external data sources are currently integrated? Average onboarding time? Is it config-driven?
27. 外部数据是否做了标准化转换（统一 JSON/字段类型）？
EN: Has external data been standardized (unified JSON, unified field types)?
28. 外部数据链路的稳定性保障机制——并发、降级熔断、查得率/错误码监控是否具备？
EN: Stability safeguards — concurrency control, circuit-breaker, hit-rate/error-code monitoring?
29. 外部数据的计费与成本管理——是否有缓存/查得率优化？单客成本可追踪吗？
EN: Billing & cost management — caching / hit-rate optimization? Is per-customer cost traceable?
30. 风险策略的入参管理——是否统一管理？映射是否可配置？加密合规是否落实？
EN: Management of risk-strategy input parameters — unified? Configurable mapping? Encryption implemented?
维度七至十一：自助分析、质量、AI、安全与业务赋能 / Dim 7-11: BI, DQ, AI, Sec & Biz Value
综合诊断意图：评估数据应用层的敏捷度、质量闭环、AI 探索阶段、云环境合规性（重点评估越南 PDPD），以及业务部门的真实体感。

维度七：自助分析与 BI

报表用什么工具？是 IT 定制开发为主，还是业务可自助拖拽分析？结果支持多端展示吗？需求积压情况如何？
EN: BI tools used? Custom-built vs Self-service? Multi-channel support? Request backlog?

维度八：数据质量与治理

是否有监控（完整性/准确性）与闭环流程（Owner/根因复盘）？主数据是否统一管理？标准是否落地执行？
EN: DQ monitoring and closed-loop process? Master data centralized? Standards enforced?

维度九：AI 与智能化赋能

算法模型落地了哪些场景？是否有"智能问数"或 RPA 数字员工？大模型（LLM）是否有探索？主要阻碍是什么？
EN: ML deployment scenarios? NL-Q&A or RPA? LLM exploration? Biggest AI blocker?

维度十：云安全、隐私合规与云成本（重点）

AWS Region 在境内还是境外？是否符合越南 PDPD（Decree 13）数据出境与本地化要求？S3 敏感数据加密与细粒度访问控制是否落地？IAM 审计与月度云成本（FinOps）现状？
EN: AWS Region? Compliant with Vietnam PDPD (Decree 13) for cross-border? S3 encryption & FGAC? IAM audit & monthly cloud FinOps?

维度十一：业务场景赋能价值验证

数据在风控（不良率改善）、营销（获客转化）、催收（催回率）、运营中发挥了多大作用？业务一号位如何打分及核心诉求是什么？
EN: Value delivered in Risk, Marketing, Collections, Ops? CEO perspective score & Top 3 desired outcomes?
